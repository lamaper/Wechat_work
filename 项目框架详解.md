# 项目框架详解

## 1. 设计原则

当前项目采用“单体后端 + 多页面前端”的实用型结构：

- 后端负责所有业务与鉴权
- 前端页面直接由后端托管
- 数据分为业务数据（SQLite）与知识数据（Markdown）

优点是部署简单、排障路径清晰。

## 2. 请求如何流转

### 页面请求

1. 浏览器访问 `/chat`、`/freshman` 等路由
2. 认证拦截器判断会话
3. 通过后返回 `app/web` 对应页面

### API 请求

1. 前端发起 `/api/...`
2. 认证模块校验权限
3. 对应业务模块处理
4. 返回 JSON

## 3. 业务模块关系

- `auth` 管“谁能访问”
- `ai` 管“怎么回答问题”
- `feature3` 管“工单和微信流程”
- `pages` 管“页面入口”
- `shared` 提供基础能力

模块之间通过函数调用协作，不做复杂事件总线。

## 4. AI 问答框架

`/api/chat` 的核心策略是：

1. 先做快速规则判断
2. 再走本地知识库检索
3. 需要时触发网页检索抓取
4. 最终让 LLM 根据证据输出

这套流程的目标是优先保证“可解释、有来源”。

## 5. 工单框架

工单围绕三类角色：

- 新生：提交与查询
- 学长：查看待答与回复
- 系统 worker：处理自动任务

状态流转保持简单：`pending -> answered -> closed`。

## 6. 扩展建议

新增功能时优先复用现有模块，避免在 backend 根目录叠加新入口文件。

- 新接口：优先放到对应模块 `routes.py + service.py`
- 新页面：放 `app/web`，再在 `modules/pages/routes.py` 挂路由
- 新数据表：统一走 `modules/shared/db.py`
