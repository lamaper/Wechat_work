# 四个功能代码框架详解

本文档基于当前仓库源码、现有开发文档与项目框架说明整理，目标不是简单重复目录结构，而是把“项目里到底有哪些功能、每个功能对应哪些代码、这些代码是如何分层协作的、改某个功能时应该从哪里入手”系统讲清楚。

本文特别面向以下场景：

- 新成员接手项目，需要快速建立“功能 -> 代码 -> 数据 -> 路由 -> 页面”的完整映射。
- 已经能运行项目，但还不清楚每个功能到底落在什么文件中。
- 需要二次开发、重构、排障、补测试，想先明确各功能边界和共享基础设施。

---

## 1. 先统一“四个功能”的理解口径

在当前仓库中，存在两套常见的理解方式：

### 1.1 按业务功能理解

如果从“用户实际能使用什么”来划分，当前项目最适合按以下四个功能理解：

1. 功能一：地点查询
2. 功能二：智能问答
3. 功能三：人工工单与微信问答
4. 功能四：登录与权限

这是本文采用的主口径，因为它最符合“一个功能对应哪些代码”的分析方式。

### 1.2 按工程模块理解

如果从后端目录结构来理解，项目被拆成以下模块：

- `modules/auth`：登录、会话、权限
- `modules/pages`：页面入口路由
- `modules/ai`：AI 问答、知识库、网页抓取
- `modules/feature3`：人工工单、微信回调、异步 worker
- `modules/shared`：数据库、路径、工具方法

这两套划分不是冲突关系，而是两个观察角度：

- “四个功能”回答的是：用户能做什么、产品能力有哪些。
- “五个模块”回答的是：代码被怎样拆开、维护时应该去哪里找。

### 1.3 为什么本文不把 `pages` 单独算作一个业务功能

`pages` 很重要，但它本质上不是独立业务能力，而是“把各个页面挂到 URL 上并施加访问控制”的入口层。也就是说：

- 地点查询真正的主体不在 `pages`，而在 `place.html`
- AI 问答真正的主体不在 `pages`，而在 `ai/*` 和 `chat.html`
- 工单真正的主体不在 `pages`，而在 `feature3/*`、`freshman.html`、`senior.html`
- 登录真正的主体不在 `pages`，而在 `auth/*` 和 `login.html`

因此本文把 `pages` 放到“共享基础设施”中说明，而不是当成单独功能。

### 1.4 一个必须记住的历史命名事实

功能三在代码里并没有被命名为 `ticket`、`workorder` 或 `service_desk`，而是沿用了历史开发阶段的名字 `feature3`。因此：

- 代码目录叫 `modules/feature3/`
- 环境变量叫 `FEATURE3_*`
- 测试脚本叫 `feature3_smoke_test.py`

这不是另一个隐藏功能，而就是当前“人工工单与微信问答”这套正式业务模块。

## 2. 项目的整体框架总览

在分析四个功能之前，先把整个项目的主框架建立起来，否则容易把页面层、路由层、服务层和数据层混在一起。

### 2.1 顶层目录如何理解

当前最重要的源码和数据目录如下：

```text
Wechat_work/
├── app/
│   ├── backend/
│   │   ├── app.py
│   │   ├── modules/
│   │   ├── tests/
│   │   ├── .env.example
│   │   ├── auth_identities.json.example
│   │   ├── official_url_allowlist.json
│   │   └── 运行期状态文件
│   ├── data/
│   └── web/
├── docs/
└── README.md
```

这几个目录分别承载不同职责：

- `app/backend/`：后端启动入口、业务模块、运行配置、测试脚本、运行状态文件。
- `app/data/`：AI 的知识库原始数据，不属于项目开发文档系统。
- `app/web/`：所有前端页面与公共前端脚本。
- `docs/`：开发、部署、测试说明文档。

### 2.2 后端启动入口的地位

整个后端只有一个真正的装配入口：[app/backend/app.py](app/backend/app.py)。

这个文件的职责非常集中：

1. 加载 `app/backend/.env`
2. 创建 Flask 应用对象
3. 把 `app/web` 作为静态目录
4. 设置 Session 相关配置
5. 初始化 SQLite 数据库
6. 同步特权身份白名单
7. 启动工单 worker
8. 注册认证路由
9. 注册页面路由
10. 注册 AI 路由
11. 注册诊断路由
12. 注册工单与微信路由

这个顺序不是随便排的，而是严格反映依赖关系：

- 数据库必须先初始化，否则工单和权限数据无处可存。
- 认证守卫必须先挂好，否则后续页面和 API 的登录控制不成立。
- worker 必须在应用启动阶段拉起，否则自动 AI 答复任务不会被消费。

### 2.3 一次典型请求会经过哪些层

不论是地点查询、AI 问答还是工单，请求通常都会经历如下层次：

1. 页面入口层：由 `modules/pages/routes.py` 把 URL 映射到静态 HTML。
2. 登录守卫层：由 `modules/auth/routes.py` 的 `before_request` 决定能否访问。
3. 业务路由层：由 `modules/auth/routes.py`、`modules/ai/routes.py`、`modules/feature3/routes.py` 接收请求参数。
4. 业务服务层：由 `modules/auth/service.py`、`modules/ai/service.py`、`modules/feature3/service.py` 实现真实业务逻辑。
5. 共享支撑层：由 `modules/shared/*` 提供数据库、路径、文本处理、ID 生成等公共能力。
6. 外部依赖层：包括 BIT 登录、腾讯地图、公开网页、LLM、微信接口等。

### 2.4 当前项目中的主要角色

当前系统里需要区分四类角色：

- 普通登录用户：可以访问首页、地点查询、AI 问答、新生工单页。
- 学长 / 答复人：在普通能力之外，还可以访问 `/senior` 处理工单。
- 管理员：拥有更高权限位，但当前没有完整后台界面。
- 微信用户：不一定登录网页，但可以通过 `/wechat` 文本消息进入工单系统。

### 2.5 共享基础设施一览

这些文件并不专属于某一个功能，但会被多个功能共同依赖：

| 文件 | 作用 | 影响到的功能 |
| --- | --- | --- |
| `app/backend/app.py` | 后端唯一装配入口 | 四个功能全部依赖 |
| `app/backend/modules/pages/routes.py` | 所有页面入口路由 | 地点查询、AI、工单、首页 |
| `app/backend/modules/shared/db.py` | SQLite 建表与 CRUD | 登录权限、工单主流程 |
| `app/backend/modules/shared/paths.py` | backend/data/web 根路径统一定义 | AI、工单、微信缓存 |
| `app/backend/modules/shared/text_utils.py` | 文本归一化、词法打分 | AI 检索与抓取评分 |
| `app/backend/modules/shared/id_utils.py` | 查询码生成 | 工单追踪 |
| `app/backend/modules/diagnostics/routes.py` | 运行状态诊断接口 | AI 与工单的运行态观察 |
| `app/web/index.html` | 统一入口页 | 用户进入各业务页面 |
| `app/web/common.js` | 前端公共方法 | 工单页、学长页、聊天页 |

## 4. 功能一：地点查询的详细框架

地点查询是当前项目里最“前端集中”的功能。它存在于统一登录系统内部，但真正的地点搜索逻辑并不在后端，而是在浏览器端直接调用腾讯地图服务。

### 4.1 功能定位

地点查询解决的问题很明确：

- 用户输入地点关键词。
- 页面调用腾讯地图地点检索接口。
- 返回地图上的位置、地点名、地址、经纬度和静态图预览。

这个功能不是“校内地点数据库查询系统”，也不是“后端地图服务封装层”。它更接近“一个受登录保护的地图检索页面”。

### 4.2 入口与访问控制

这个功能的入口链路是：

1. 用户访问首页 `/`。
2. 首页 [app/web/index.html](app/web/index.html) 提供“进入地点查询”的入口按钮。
3. 后端 [app/backend/modules/pages/routes.py](app/backend/modules/pages/routes.py) 把 `/place` 映射到 [app/web/place.html](app/web/place.html)。
4. 在真正返回页面之前，[app/backend/modules/auth/routes.py](app/backend/modules/auth/routes.py) 的登录守卫会判断是否已登录。
5. 未登录则跳转到登录页，已登录才会进入地点查询页。

因此，地点查询虽然几乎没有后端业务逻辑，但仍然完全受统一登录系统保护。

### 4.3 代码层结构

#### 4.3.1 页面主体：`app/web/place.html`

这是地点查询真正的主体文件，负责：

- 引入腾讯地图 JavaScript API。
- 初始化地图对象 `TMap.Map`。
- 初始化标记层 `TMap.MultiMarker`。
- 监听按钮点击和回车提交。
- 通过 JSONP 调用腾讯地图地点搜索接口。
- 解析搜索结果并更新：
  - 地图中心
  - 标记点
  - 地点名称
  - 地址
  - 经纬度
  - 静态预览图

#### 4.3.2 页面入口：`app/backend/modules/pages/routes.py`

这个文件并不做地点查询计算，只负责把 `/place` 返回为 `place.html`。它的意义在于：

- 把地点页纳入统一站点结构。
- 让地点页共享整套登录态保护。

#### 4.3.3 认证保护：`app/backend/modules/auth/routes.py`

地点页不是公开页，因此必须经过登录守卫。这个守卫会在所有请求进入业务前判断：

- 当前路径是否属于公开路径。
- 当前 session 中是否已有登录用户。

若无登录态，访问 `/place` 时不会得到页面，而是跳去 `/login`。

### 4.4 页面内部执行流程

从用户点击“查询地点”开始，这个功能的实际执行步骤是：

1. 页面读取输入框中的关键词。
2. 检查腾讯地图 Key 是否仍是占位符。
3. 如果 Key 未替换，直接提示无法查询。
4. 如果关键词为空，提示用户先输入关键词。
5. 构造腾讯地图 WebService 的 JSONP URL。
6. 动态插入 `<script>` 发起 JSONP 请求。
7. 腾讯地图返回结果后，页面回调函数解析 `res.data[0]`。
8. 提取地点的 `title`、`address`、`location.lat`、`location.lng`。
9. 地图重设中心点和缩放级别。
10. 标记层更新为新地点。
11. 详情卡片更新名称、地址、坐标。
12. 静态图 URL 重新生成并替换预览图。

### 4.5 为什么这个功能没有独立后端模块

因为当前代码设计里，地点查询并不需要你自己的后端参与搜索逻辑。搜索服务由腾讯地图提供，浏览器直接请求它即可。这样设计的结果是：

- 优点：实现简单、部署轻、页面即开即用。
- 缺点：功能可用性高度依赖腾讯地图 Key、网络、域名限制和调用额度。

换句话说，地点查询在当前项目中不是“后端算法功能”，而是“前端集成功能”。

### 4.6 外部依赖与配置边界

地点查询真正依赖的是腾讯地图配置，而不是本地数据库或后端服务逻辑。

当前至少依赖以下条件：

- `place.html` 中的 `__TENCENT_MAP_KEY__` 已被真实 Key 替换。
- 腾讯位置服务控制台已开通相关接口权限。
- 域名来源限制允许当前访问域名。
- 当日调用量没有超限。

这也是为什么这个功能最常见的问题不是 Python 报错，而是：

- Key 无效
- 配额已达上限
- 服务未开通
- 域名不在白名单内

### 4.7 当前限制与后续扩展点

当前地点查询的限制主要有：

- 没有后端缓存或代理层。
- 没有地点收藏、历史记录、校区分类等业务数据层。
- 没有把地图能力封装成内部 API。

如果以后要扩展，最自然的方向是：

- 为校内高频地点增加预置列表或别名。
- 增加校区、楼宇、办事点等结构化分类。
- 将前端直接调腾讯接口改为后端代理，以便统一控制配额与日志。

## 5. 功能二：智能问答的详细框架

智能问答是当前项目中代码最复杂、链路最长、外部依赖最多的功能。它不是一个“只调大模型”的简单聊天页，而是一套“高频问题缓存 + 本地知识库检索 + 公开网页抓取 + 可选 LLM 生成”的混合问答系统。

### 5.1 功能定位

这个功能要解决的不是开放域聊天，而是“围绕北理工新生问题给出相对可信、可落地、可追溯的回答”。

因此它的设计重点不是“尽量像通用聊天机器人”，而是：

- 优先命中学校相关知识。
- 尽量保留来源。
- 在证据不足时能够回退或明确说明不确定。
- 在外部网络条件不佳时依然能靠本地知识库工作。

### 5.2 前端页面层：`app/web/chat.html`

这个页面负责：

- 渲染聊天界面。
- 维护用户输入框、发送按钮、示例问题按钮。
- 调用 `POST /api/chat`。
- 把返回的 `answer` 当作 Markdown 渲染成最终回答。
- 根据后端返回的 `mode` 更新界面状态。

这个页面的设计特点是“前端克制、后端集中”。也就是说：

- 页面不自行决定回答策略。
- 页面不自己做知识检索。
- 页面只负责交互和渲染。

因此 AI 功能的复杂度基本都在后端。

### 5.3 路由层：`app/backend/modules/ai/routes.py`

这个文件提供四个主要接口：

- `POST /api/chat`：主问答接口
- `POST /api/rebuild_index`：重建知识库索引
- `GET /api/kb_query`：直接查看知识库检索结果
- `GET /api/demo_status`：查看 AI 与工单的运行态

它的职责非常明确：

- 校验参数是否为空。
- 做管理员权限判断（如重建索引）。
- 调用 service 层。
- 把结果封装为 JSON 返回。

真正的 AI 处理并不在这里，而在 `service.py`。

### 5.4 业务编排层：`app/backend/modules/ai/service.py`

这是 AI 功能的核心中枢文件。它把多个子能力串成一条完整回答链路。

#### 5.4.1 文件中承担的主要职责

该文件主要承担以下职责：

- 读取 FAQ 数据
- 维护高频问题缓存
- 调用知识库查询
- 调用网页抓取回答链路
- 调用 LLM
- 统一组装最终回答
- 生成运行态摘要信息

#### 5.4.2 高频问题缓存

`HIGH_FREQ_CACHE` 用于承载一些非常常见、模式相对固定的问题，例如：

- 选课
- 校园网
- 宿舍

当问题中包含这些高频关键词时，可以直接返回预置答案，而不必走更昂贵的知识库或网页抓取流程。

这条路径的价值在于：

- 响应更快
- 结果更稳定
- 对外部依赖更少

#### 5.4.3 FAQ 与知识库检索

`retrieve_from_faq()` 的职责并不是只查 `faq.md`。它的真实逻辑顺序是：

1. 先尝试高频缓存命中。
2. 再调用 `query_index()` 查本地知识库索引。
3. 如果索引没有结果，再回退到 `faq.md` 的问答对匹配。

因此它本质上是“本地知识命中入口”，而不是狭义 FAQ 查询器。

#### 5.4.4 LLM 调用

`service.py` 同时支持：

- DeepSeek
- OpenAI 兼容接口

LLM 的地位是“基于已检索证据组织语言”，而不是“直接自由生成答案”。

系统提示词明确约束：

- 只能基于给定参考知识回答
- 如果证据不足要明确说明
- 需要在答案末尾列来源

这说明该项目的 AI 设计目标是“受约束的校园问答”，不是开放生成。

#### 5.4.5 主回答组装逻辑：`build_answer()` 与 `build_chat_response()`

这两个函数是最需要理解的地方。

`build_answer()` 更偏向“基于本地知识体系生成回答”，会处理：

- 高频缓存直接命中
- LLM 基于检索知识生成回答
- 无 LLM 时直接返回检索结果与兜底文案

`build_chat_response()` 则是更外层的“总调度器”，负责在本地知识库和网页抓取之间做选择，并返回结构化结果：

- `answer`
- `steps`
- `web_sources`
- `mode`

#### 5.4.6 一个必须注意的真实行为

虽然文档常写“网页抓取优先，知识库回退”，但当前源码的真实逻辑更精确地说是：

- 先做本地知识预检
- 如果本地知识分数已经足够高，则直接走知识库回答
- 只有本地证据不够强时，才继续网页搜索和抓取

这意味着当前代码追求的不是“永远先搜网页”，而是“优先使用更便宜、更稳定、足够可信的本地知识”。

### 5.5 知识库与索引层：`app/backend/modules/ai/kb.py`

这个文件负责把 `app/data/` 中的 Markdown 知识文档转成可检索结构。

#### 5.5.1 数据来源

AI 的主要本地数据不在 `docs/`，而在 `app/data/`。这些文件不是开发文档，而是运行时知识语料，例如：

- `faq.md`
- `bit_freshman_rag_handbook.md`
- 入学准备、教务服务、住宿网络、奖助学金、校园安全等专题 Markdown

#### 5.5.2 处理流程

`kb.py` 的核心处理流程是：

1. 遍历 `app/data/*.md`
2. 按 Markdown 标题和段落切成逻辑块
3. 再按固定块大小和重叠量切成 chunk
4. 保存 `source`、`chunk_id`、`text`
5. 构建词法索引或可选向量索引

#### 5.5.3 为什么有词法模式和向量模式

当前项目支持两套检索后端：

- 词法检索：默认更稳，不依赖外网模型下载
- FAISS 向量检索：效果潜力更高，但依赖 `sentence-transformers` 与模型下载环境

配置项 `KB_FORCE_LEXICAL` 的意义就在这里：

- 为 `1` 时，直接强制使用词法模式
- 为 `0` 时，优先尝试向量检索，失败再回退

#### 5.5.4 运行时状态文件

索引构建后会在 `app/backend/` 下留下这些运行态文件：

- `kb_meta.json`
- `kb_texts.json`
- `kb_state.json`
- 可选 `kb_index.faiss`

这些文件不是源码结构的一部分，而是索引构建后的运行期产物。

### 5.6 网页搜索与抓取层：`app/backend/modules/ai/web_fetcher.py`

这是 AI 模块里另一个很重的文件。它不是简单 `requests.get()` 几个网页，而是做了一整套受约束的网页证据获取流程。

#### 5.6.1 它承担的事情

主要包括：

- 构造搜索查询词
- 并行向多个搜索源取候选链接
- 清理和标准化候选 URL
- 过滤不安全或低价值链接
- 对候选链接做打分排序
- 检查公网安全边界
- 检查 `robots.txt`
- 获取 HTML 并抽取正文
- 生成网页证据摘要和来源列表

#### 5.6.2 安全边界

它明确拒绝访问以下类型的地址：

- 本机地址
- 内网地址
- 私有保留地址
- 不规范 URL

所以某些抓取失败不一定代表 bug，也可能是安全策略正常生效。

#### 5.6.3 站点优先级控制

站点偏好配置在 [app/backend/official_url_allowlist.json](app/backend/official_url_allowlist.json)。

这个文件把网站分成两类：

- `sites`：优先信任、优先抓取的学校官方站点
- `restricted_sites`：即便是校内相关站点，也因为需要登录或匿名价值很低而不应作为抓取目标的站点

例如：

- `www.bit.edu.cn`、`jwc.bit.edu.cn`、`hi.bit.edu.cn` 属于优先站点
- `ehall.bit.edu.cn`、`sso.bit.edu.cn`、`webvpn.bit.edu.cn` 属于受限站点

#### 5.6.4 为什么 AI 要保留网页抓取

因为本地知识库再完整，也不可能实时覆盖所有新通知、新公告、新页面更新。网页抓取的价值在于：

- 补充知识库中没有的新信息
- 优先回答“最新通知入口”这类实时性较强的问题
- 让系统能在部分场景下利用官网实时页面作为证据

### 5.7 AI 功能的一次完整请求流

从用户在聊天页发问到得到回答，大致会经历如下过程：

1. `chat.html` 提交问题到 `POST /api/chat`
2. `ai/routes.py` 检查问题非空
3. `build_chat_response()` 先做本地知识预检
4. 如果本地知识分数高，直接走知识库回答
5. 如果本地知识不足，进入 `search_and_fetch_public_web()`
6. 若抓到可用网页，则生成网页证据块
7. 若配置了 LLM，则基于证据生成自然语言回答
8. 若未配置 LLM，则输出检索/抓取摘要兜底
9. 最终以 `answer + mode + steps + web_sources` 返回前端
10. `chat.html` 渲染回答并更新“知识库模式 / 网页模式”的状态提示

### 5.8 诊断与测试

AI 功能目前的运行态观察口主要有两个：

- `GET /api/demo_status`
- `POST /api/rebuild_index`

自动化测试主要是 [app/backend/tests/url_fetch_smoke_test.py](app/backend/tests/url_fetch_smoke_test.py)。它重点验证的不是聊天 UI，而是 AI 链路里最容易出问题的“搜索与抓取支撑能力”。

### 5.9 当前限制与维护重点

AI 功能最重要的限制有：

- 外部网页抓取受网络环境影响明显
- 向量索引依赖模型下载和额外安装条件
- 回答质量同时取决于知识库内容、站点质量、LLM 可用性
- 前端当前只展示最终回答，没有完整暴露 `steps` 调试信息

维护这个功能时，最需要优先分清三条路径：

- 本地高频缓存路径
- 本地知识库路径
- 网页抓取路径

如果不先分清这三条路径，很容易误判“为什么这次没有走网页搜索”或“为什么没调到 LLM”。

## 6. 功能三：人工工单与微信问答的详细框架

人工工单与微信问答是当前仓库里最像“完整业务系统”的模块。它不是一个简单的提交表单，而是一套带角色、状态机、异步任务和微信入口的轻量工单系统。

### 6.1 功能定位

该功能解决的问题是：

- AI 不能很好回答的问题，需要转给人工处理。
- 新生提交问题后，需要有稳定的进度查看方式。
- 学长或答复人需要有待办处理台。
- 微信用户即使没有网页登录，也能把文本消息纳入同一套问题体系。

### 6.2 这个功能为什么在代码里叫 `feature3`

这是当前代码中一个很重要的历史遗留事实。功能三最初按阶段开发时就使用了 `feature3` 命名，后续虽然功能已经正式化，但目录名和环境变量名没有改掉。

所以当前看到这些名字时，要知道它们全部都在指“人工工单与微信功能”：

- `modules/feature3/`
- `FEATURE3_*`
- `feature3.db`
- `feature3_smoke_test.py`

### 6.3 前端页面层

#### 6.3.1 新生页：`app/web/freshman.html`

这个页面主要负责四件事：

- 提交新问题
- 查看当前登录账号下的问题列表
- 通过“问题编号 + 查询码”查询问题状态
- 对已答复问题执行关闭操作

此外，它还做了不少前端体验逻辑：

- 自动轮询“我的问题”
- 把新回复高亮显示
- 用 localStorage 保存已读答复时间
- 支持主动清除提醒

这说明该页面并不是一个纯静态表单，而是具备轻量前端状态管理。

#### 6.3.2 学长页：`app/web/senior.html`

这个页面主要负责：

- 展示待处理问题列表
- 展示已处理问题列表
- 搜索待办和历史问题
- 提交人工答复

这个页面的特点是：

- 前端显示逻辑相对简单
- 核心业务约束全部放在后端权限体系里

### 6.4 路由层：`app/backend/modules/feature3/routes.py`

这个文件承担的是“对外接口层”，负责把不同的工单请求分流给 service 和 db 层。

#### 6.4.1 新生侧接口

- `POST /api/questions`：提交问题
- `GET /api/questions/mine`：查看当前账号名下的问题
- `GET /api/questions/track`：按编号和查询码查单
- `PUT /api/questions/<id>/close`：关闭问题

#### 6.4.2 学长侧接口

- `GET /api/tasks/pending`：查看待处理问题
- `GET /api/tasks/answered`：查看已处理问题
- `POST /api/answers`：提交人工答复

#### 6.4.3 运行态接口

- `GET /api/feature3_status`：返回 worker、任务汇总、当前用户权限等状态

#### 6.4.4 微信回调接口

- `GET /wechat`：微信服务器验签
- `POST /wechat`：接收微信文本消息

### 6.5 业务服务层：`app/backend/modules/feature3/service.py`

这是工单系统真正的核心业务文件。

#### 6.5.1 它负责的事情

主要包括：

- worker 生命周期管理
- 任务入队与消费
- AI 自动答复任务处理
- 微信文本命令解析
- 微信签名校验
- 工单创建后的被动回复文本构建
- 工单运行态摘要构建

#### 6.5.2 任务系统与 worker

这个模块没有引入 Celery 之类的独立任务队列，而是直接在同一 Flask 进程里启动后台线程。

启动时：

- `app.py` 调用 `start_feature3_worker(app)`
- worker 线程进入 `_worker_loop()`
- 循环从 `tasks` 表中 `claim_next_task()` 领取任务
- 调用 `_process_task()` 执行
- 成功则 `mark_task_done()`
- 失败则 `mark_task_retry()`，必要时标记为失败

这种设计的优点是：

- 本地开发简单
- 单实例部署方便
- 不需要额外消息队列

缺点是：

- 高并发能力有限
- 多实例部署复杂
- 与 Web 请求共享同一进程资源

#### 6.5.3 AI 自动答复能力

如果开启 `FEATURE3_AUTO_AI_ANSWER`，新问题创建后会调用 `enqueue_ai_answer_if_enabled()` 把任务写入 `tasks` 表。

worker 取到任务后会：

1. 根据 `question_id` 读取问题
2. 确认问题仍是 `pending`
3. 调用 AI 模块的 `build_answer()` 生成答复
4. 把答复以 `answer_type="ai"` 保存到 `answers` 表

也就是说，工单模块和 AI 模块之间的耦合点主要在这里。

#### 6.5.4 微信文本逻辑

微信文本的处理分两种路径：

- 如果用户发的是“查 编号 查询码”格式，则解析为状态回查命令。
- 如果用户发的是普通问题，则创建新工单并返回“问题编号 + 查询码 + 页面提示”。

这个设计非常关键，因为它把微信公众号文本消息和网页工单流统一到了同一个问题表结构上。

### 6.6 数据层：`app/backend/modules/shared/db.py`

工单系统的持久化核心在 `shared/db.py`，而不是在 `feature3` 目录里单独再写一个数据库文件。

#### 6.6.1 主要表结构

当前与工单关系最紧密的表包括：

- `users`
- `questions`
- `answers`
- `tasks`
- `auth_identities`

#### 6.6.2 `users` 表

这不是传统意义上的“登录账号表”，而是系统中的“业务身份实体表”。这里面既可以存：

- 登录用户的 `owner_identity`
- 学长答复人的身份
- 微信用户映射出来的外部身份

#### 6.6.3 `questions` 表

记录问题主体，包括：

- 提问者 ID
- 标题
- 内容
- 状态
- 查询码
- 渠道
- 创建/更新时间

#### 6.6.4 `answers` 表

记录问题对应的最终答复，包括：

- 问题 ID
- 答复人 ID
- 答复类型（人工 / AI）
- 答复内容
- 创建/更新时间

#### 6.6.5 `tasks` 表

记录异步任务队列状态，包括：

- 任务类型
- 载荷 payload
- 当前状态
- 尝试次数
- 最大重试次数
- 最近错误信息

#### 6.6.6 `auth_identities` 表

虽然它不属于工单业务本体，但它决定谁能进入学长处理台，因此仍是工单权限体系的重要支撑表。

### 6.7 状态机：这个功能的真正核心

工单系统最核心的不是表单，而是状态机。

当前主状态有三个：

- `pending`
- `answered`
- `closed`

典型流转是：

1. 用户提问后创建 `pending`
2. 学长或 AI 给出答复后变为 `answered`
3. 发起者确认结束后变为 `closed`

这意味着工单模块的真实重点是“状态流转是否正确”，而不是“能否写入一条记录”。

### 6.8 网页提问链路

网页登录态下的工单流程如下：

1. 登录用户进入 `/freshman`
2. 提交 `POST /api/questions`
3. 后端从当前登录用户中取 `owner_identity`
4. `get_or_create_user()` 确保业务身份存在
5. `create_question()` 写入问题表，默认状态为 `pending`
6. 如果启用自动 AI 回复，则入队异步任务
7. 前端通过 `/api/questions/mine` 查看自己的问题列表
8. 已回答问题可通过 `PUT /api/questions/<id>/close` 关闭

### 6.9 学长处理链路

学长处理台的流程如下：

1. 有 `PERM_ANSWER` 权限的用户访问 `/senior`
2. 页面调用 `/api/tasks/pending` 获取待办列表
3. 学长填写回复并提交到 `/api/answers`
4. 后端校验该问题存在且未关闭
5. 后端为学长创建或获取 `priv:<student_code>` 形式的答复人身份
6. `save_answer()` 写入答案并把问题状态置为 `answered`
7. 页面刷新待办与已处理列表

### 6.10 微信进入链路

微信用户与网页用户最大的不同是：

- 微信用户可以不登录网页
- 其身份是通过外部标识映射到 `users` 表

一条微信文本进入后大致如下：

1. 微信服务器调用 `POST /wechat`
2. 后端先做签名校验
3. 解析 XML 中的 `MsgType`、`FromUserName`、`Content`
4. 若消息不是文本，则简单返回 success
5. 若文本是查单命令，则按编号和查询码查问题
6. 若文本是新问题，则创建微信来源工单
7. 返回被动回复 XML 给微信服务器

### 6.11 工单功能与登录系统的耦合点

工单模块最重要的耦合点不是学号，而是 `owner_identity`。它来自认证模块的 `build_owner_identity()`。

这个设计的价值是：

- “我的问题”不必直接依赖明文学号
- 业务归属稳定可复现
- 权限系统和工单系统可以共享统一身份语义

### 6.12 微信主动推送的当前位置

当前代码中，答复提交成功后还预留了“微信主动客服消息推送”的逻辑，辅助文件在 [app/backend/modules/feature3/wechat_utils.py](app/backend/modules/feature3/wechat_utils.py)。

它负责：

- 读取/缓存 `access_token`
- 调用微信客服消息接口
- 遇到 token 失效时自动重试刷新

需要注意的是，这条路径当前属于“可选增强能力”，不是默认主链路。

### 6.13 测试与维护重点

工单功能的自动化核心测试是 [app/backend/tests/feature3_smoke_test.py](app/backend/tests/feature3_smoke_test.py)。

它覆盖的关键动作包括：

- 创建问题
- 查看我的问题
- 学长回复
- 按编号和查询码追踪
- 已回答列表搜索
- 关闭问题
- 验证关闭后不能再次回复

这说明工单模块当前最重要的维护重点不是页面样式，而是：

- 状态是否正确流转
- 权限是否正确限制
- 关闭后是否还能被错误修改
- 微信与网页是否使用同一套持久化语义

## 7. 功能四：登录与权限的详细框架

登录与权限是整个系统的基础层。地点查询、AI、工单页面基本都建立在它之上；没有这层，项目就只是几张静态页和一组匿名接口。

### 7.1 功能定位

当前登录功能的目标不是做一个复杂的用户中心，而是提供一套足够稳定的统一访问控制机制：

- 普通用户必须先登录才能使用主要页面和 API
- 少量学长或管理员可以附加更高权限
- 登录信息与工单归属保持一致

### 7.2 页面层：`app/web/login.html`

登录页负责：

- 展示账号/密码输入框
- 调用 `POST /api/auth/login`
- 读取 URL 中的 `next` 参数
- 登录成功后跳转到 `next` 或首页
- 展示登录失败或网络失败提示

需要注意的是，这个页面并不自己实现权限逻辑，它只是登录入口。

### 7.3 路由层：`app/backend/modules/auth/routes.py`

这个文件有两类职责：

#### 7.3.1 认证相关接口与页面

- `GET /login`
- `GET /logout`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

#### 7.3.2 全局登录守卫

通过 `@app.before_request` 挂接登录守卫，决定：

- 哪些路径是公开的
- 哪些路径必须已登录
- 页面和 API 在未登录时应返回什么

未登录时：

- 页面请求：重定向到 `/login?next=...`
- API 请求：返回 `401` JSON

这层逻辑是整个系统统一门禁的核心。

### 7.4 业务服务层：`app/backend/modules/auth/service.py`

这是登录与权限功能真正的核心文件。

#### 7.4.1 它负责的主要事情

- 调用 BIT 登录依赖完成统一身份认证
- 把外部用户资料转换为当前系统的 session 用户对象
- 定义权限位
- 判断当前用户是否已登录、是否有某项权限
- 构造登录所需的 `owner_identity` 和外部身份标识
- 同步本地特权身份白名单

#### 7.4.2 BIT 登录接入

`authenticate_bit_user()` 会调用 `bit_login` 库，完成：

- 登录
- 取学生资料
- 抽取学号、姓名、专业、学院、年级、班级等字段

这说明当前项目的真实登录认证并不在本地数据库，而在外部 BIT 统一身份系统。

#### 7.4.3 Session 用户对象

登录成功后，系统不会只在 session 里存一个简单的用户名，而是存一个结构化用户对象，包含：

- `student_code`
- `name`
- `major`
- `college`
- `grade`
- `class_name`
- `permissions`
- `permission_labels`
- `owner_identity`
- `is_privileged`

这意味着 session 不只是“已登录”的标记，也是后续页面和工单逻辑的共享上下文。

#### 7.4.4 权限位模型

当前定义了三个权限位：

- `PERM_USE_APP`：基础使用权限
- `PERM_ANSWER`：学长答复权限
- `PERM_ADMIN`：管理员权限

普通登录用户默认只会得到基础使用权限。答复权限和管理员权限不是自动给所有人，而是通过本地特权身份附加。

#### 7.4.5 `owner_identity` 的地位

这是整个登录功能里最关键的一个内部字段。它由 `build_owner_identity(student_code)` 基于私有盐和学号生成哈希字符串。

它的作用不是显示给用户看，而是作为系统内部的“稳定业务归属 ID”，主要用于：

- 工单归属绑定
- `/api/questions/mine` 查当前用户问题
- 避免在业务表中直接依赖明文学号

可以说，没有 `owner_identity`，登录和工单之间就缺了一条最重要的桥。

#### 7.4.6 外部身份映射

除了网页登录用户，系统还需要支持微信外部用户，因此 `build_external_identity(channel, external_id)` 用于生成外部身份标识。这使得：

- 微信用户可以进入同一套工单数据体系
- 外部渠道身份不需要复用网页登录学号语义

### 7.5 特权身份机制

当前系统并没有给所有登录用户做完整的权限后台，而是采用“少量特权身份白名单”的设计。

相关配置样例在 [app/backend/auth_identities.json.example](app/backend/auth_identities.json.example)。

它大致表达的是：

- 某个学号可以附加 `answer`
- 某个学号可以附加 `admin`
- 启动时这些配置会被同步到 `auth_identities` 表

这种方式的优点是：

- 简单
- 足够支撑当前阶段
- 不需要额外做复杂后台

缺点是：

- 权限管理仍偏手工
- 不适合大规模角色管理

### 7.6 与页面路由的协同

登录与权限不会单独决定所有页面如何返回，而是和 `pages/routes.py` 协同工作：

- `auth/routes.py` 负责全局登录要求
- `pages/routes.py` 负责页面本身映射关系
- `/senior` 由 `pages/routes.py` 再额外调用 `require_permission(PERM_ANSWER)` 做页面级二次保护

也就是说，权限保护不是只挂在 API 上，而是页面和 API 双层都做了限制。

### 7.7 当前限制与维护重点

登录功能最需要注意的边界有：

- BIT 登录依赖外部服务与第三方库安装环境
- session 方案天然更偏传统单体 Web，不是前后端分离 token 模式
- 权限管理目前仍是轻量白名单模型
- 工单系统对 `owner_identity` 的依赖很强，不能轻易改动这部分语义

如果以后要扩展“正式后台管理”，最自然的方向是：

- 做权限管理界面
- 做学长/管理员身份配置界面
- 做权限变更审计

## 8. 页面导航层与共享基础设施的详细说明

虽然本文按四个业务功能展开，但如果不把共享层讲清楚，后续维护时仍然会找不到入口。

### 8.1 页面入口层：`app/backend/modules/pages/routes.py`

这个文件负责把静态页面挂到具体路径上：

- `/` -> `index.html`
- `/chat` -> `chat.html`
- `/place` -> `place.html`
- `/freshman` -> `freshman.html`
- `/senior` -> `senior.html`

它的作用不是“做页面内容”，而是：

- 统一页面入口
- 把静态页纳入后端权限控制
- 为后续新增页面提供统一接入点

### 8.2 首页：`app/web/index.html`

首页不是单纯展示页，而是站点级导航总入口。它至少承担以下职责：

- 展示当前登录用户名称
- 提供地点查询入口
- 提供智能问答入口
- 提供人工工单入口
- 提供学长处理台入口
- 提供退出登录入口

因此它虽然不是“业务实现核心”，但它是用户理解整个系统能力的第一入口。

### 8.3 前端公共工具：`app/web/common.js`

这个文件当前非常轻，只提供：

- `escapeHtml()`：防止前端渲染时注入 HTML
- `formatUnixTimestamp()`：把后端 Unix 时间戳格式化为本地可读时间

它虽然很小，但说明前端已经开始抽取公共逻辑，而不是把相同代码散落在多个页面里。

### 8.4 共享路径定义：`app/backend/modules/shared/paths.py`

该文件统一定义：

- `BACKEND_ROOT`
- `APP_ROOT`
- `DATA_ROOT`
- `WEB_ROOT`

它的意义是避免各模块自己用相对路径乱拼，导致：

- 启动目录变了就读不到文件
- 测试运行路径和正式运行路径不一致

### 8.5 文本工具：`app/backend/modules/shared/text_utils.py`

该文件虽然不大，但对 AI 模块非常关键。它负责：

- 文本归一化
- 生成字符 n-gram
- 计算词法相似度

AI 的 FAQ 匹配、知识库词法检索、网页候选评分都依赖这些基础文本方法。

### 8.6 查询码工具：`app/backend/modules/shared/id_utils.py`

该文件只提供一件事：生成查询码。

它看起来很小，但实际上它是工单“编号 + 查询码”追踪模式的关键基础。没有它，公众号查单和用户凭码追踪这条路径就无法成立。

### 8.7 诊断接口：`app/backend/modules/diagnostics/routes.py`

该文件当前提供 `/api/demo_status`，会拼装：

- AI 运行态摘要
- 工单运行态摘要
- 关键路由映射

它不是最终用户功能，但对开发、联调、排障非常重要，因为它能快速告诉你：

- LLM 有没有启用
- 知识库索引是否 ready
- 工单 worker 是否存活
- 当前系统哪些页面/API 预期存在

## 10. 四个功能之间的关系与依赖

理解每个功能之后，还需要再往上看一层：四个功能不是孤立的。

### 10.1 登录与权限是其他功能的前置条件

地点查询、AI 问答、新生工单、学长处理台几乎都依赖登录态。登录模块因此不是并列附属功能，而是平台基础层。

### 10.2 页面导航层是所有功能的统一入口壳

地点页、AI 页、工单页虽然业务不同，但它们先统一挂在 `pages/routes.py` 下，再接受登录守卫控制。

### 10.3 工单功能与 AI 功能存在有限耦合

两者最重要的耦合点是“自动 AI 答复任务”：

- 工单模块负责入队、取任务、落答复
- AI 模块负责生成回答文本

但 AI 模块并不主动修改工单状态，而是通过工单 service/db 层完成落库。这是一种相对清晰的边界。

### 10.4 AI 与工单共享不同类型的基础设施

- AI 更依赖文本工具、路径工具、知识数据、抓取站点配置。
- 工单更依赖数据库、ID 生成、权限身份、worker 任务表。

### 10.5 地点查询和其他功能共享的是“站点框架”，不是业务核心

地点查询和其他功能共享：

- 登录态
- 首页导航
- 页面入口路由

但不共享复杂的后端业务逻辑。它更像一个嵌入在统一系统里的外部服务前端页面。

## 12. 总结：如何用一张脑图理解当前项目

如果只保留一套最核心的理解方式，可以这样看当前项目：

### 12.1 最上层：用户可见的四个功能

1. 地点查询
2. 智能问答
3. 人工工单与微信
4. 登录与权限

### 12.2 中间层：支撑这四个功能的工程模块

- `auth`
- `pages`
- `ai`
- `feature3`
- `shared`

### 12.3 最底层：运行所依赖的外部条件

- BIT 登录服务
- 腾讯地图服务
- LLM 接口
- 公开网页搜索与抓取环境
- 微信公众号接口
- SQLite 文件与运行态文件

### 12.4 当前项目最重要的三个事实

第一，地点查询几乎是纯前端集成能力，不是独立后端业务模块。  
第二，智能问答不是单一路径，而是高频缓存、本地知识、网页抓取、LLM 组合而成的混合系统。  
第三，人工工单与微信功能虽然历史上叫 `feature3`，但现在已经是项目里最完整、最正式的一条业务状态机。  

如果把这三点理解透，再去看 `app.py`、`auth`、`ai`、`feature3`、`pages` 这几层，就能比较稳定地读懂当前整个项目。
