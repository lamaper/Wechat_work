## 一、确认存在的错误

### 1.1 【中】静态资源被登录拦截，登录页面无样式

**位置**: [modules/auth/routes.py:13-22](app/backend/modules/auth/routes.py#L13-L22)

**现象**: 未登录用户访问 `/theme.css` 和 `/common.js` 时返回 HTTP 302 重定向到登录页，导致登录页面本身无法加载 CSS 和 JS。

**原因**: `_public_path()` 函数的白名单只包含 `/login`、`/logout`、`/api/auth/`、`/wechat`、`/favicon`，未包含 `/theme.css`、`/common.js` 等静态资源路径。

**验证结果**:
```
$ curl -s -o /dev/null -w "%{http_code}" http://localhost:5000/theme.css
302
$ curl -s -o /dev/null -w "%{http_code}" http://localhost:5000/common.js
302
```

**影响**: 未认证用户看到的登录页面无任何样式，体验较差。

**修复方向**: 在 `_public_path()` 中增加对 `.css`、`.js` 等静态资源扩展名的豁免，或将 `static_folder` 路径加入白名单。

---

### 1.2 【低】`auth_identities.json` 缺失

**位置**: [modules/auth/service.py:28](app/backend/modules/auth/service.py#L28)

**现象**: 只有 `auth_identities.json.example`（示例文件），没有实际的 `auth_identities.json`。系统启动时 `sync_auth_identities_from_file()` 因为文件不存在而返回 0，不会报错，但也无法配置任何管理员或答复人。

**影响**: 
- 没有用户拥有 `PERM_ANSWER` 权限（无法进入学长处理台 `/senior`）
- 没有用户拥有 `PERM_ADMIN` 权限（无法调用 `/api/rebuild_index`）
- 只能通过直接操作 SQLite 数据库添加权限

**修复方向**: 
- 部署时从 `.example` 模板创建 `auth_identities.json` 并填入实际学号
- 或将权限管理改为数据库 UI 操作

---

### 1.3 【低】`theme.css` 存在两段重复的 `:root` 变量声明

**位置**: [theme.css:1-32](app/web/theme.css#L1-L32)（第一段，被覆盖）和 [theme.css:809-833](app/web/theme.css#L809-L833)（第二段，实际生效）

**现象**: 第一段 `:root` 定义了暖红色系（`--accent-sage: #951b31`），第二段 `:root` 覆盖为冷绿色系（`--accent-sage: #7f9186`）。第一段的所有变量从未生效，属于死代码。

**影响**: 无功能影响，但增加了文件体积和后续维护者的困惑。

**修复方向**: 删除第一段 `:root` 声明（第 1-32 行），或将第二段作为覆盖层并保留第一段作为 fallback（但当前 CSS 结构不支撑此用法）。
