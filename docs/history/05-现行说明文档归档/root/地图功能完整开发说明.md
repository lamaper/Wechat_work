# 地图功能完整开发说明

本文档记录当前地图功能的完整实现方式、代码入口、关键函数、测试结论、已知边界和后续扩展思路。它面向维护者和后续迭代者，目标是让你能快速定位地图功能是如何挂载、如何查询、如何兜底、以及如何排查问题。

## 1. 功能目标

地图页的目标不是做一个通用地图站点，而是做一个面向北京理工大学场景的地点查询页。当前页面需要同时满足以下能力：

- 搜索校园内常用地点，并展示候选结果。
- 优先返回校内可用点位，避免把用户带到校外同名地点。
- 当腾讯在线检索没有结果时，自动回退到本地校园地点库。
- 支持最近搜索、校园快捷地点、候选切换、复制坐标、复制地址、打开腾讯地图外链。
- 在腾讯静态图加载失败时，提供可读的本地预览占位图。

## 2. 地图相关代码文件

地图功能直接相关的代码文件只有少数几个，核心逻辑都集中在前端页面里。

- [app/backend/app.py](app/backend/app.py#L20)：Flask 入口文件，负责加载并注册页面路由、认证路由、AI 路由、诊断路由和工单路由。地图页本身不是单独 API，而是由这里挂载进应用。
- [app/backend/modules/pages/routes.py](app/backend/modules/pages/routes.py#L6)：页面路由注册点，定义了 /place 对应 place.html，并通过 send_from_directory 从静态目录返回页面。
- [app/web/place.html](app/web/place.html#L890)：地图页主文件。页面结构、样式引用、腾讯地图 JS API 加载、WebService 搜索、本地 POI 兜底、候选结果渲染、历史记录和交互事件都在这里。
- [app/web/theme.css](app/web/theme.css#L841)：共享视觉样式文件。地图页继承了其中的按钮、卡片、历史 chip、快捷地点、结果项、空状态等样式。

补充说明：地图页当前不依赖 [app/web/common.js](app/web/common.js) 的通用脚本，所有地图专属交互都在 place.html 内部自洽完成。

## 3. 页面结构

地图页 HTML 结构从 [app/web/place.html](app/web/place.html#L890) 开始。页面主体分成几块：

- 顶部 hero 区：标题、返回首页、智能问答、退出登录。
- 搜索工具栏：关键词输入框、查询按钮、清空结果按钮。
- 校园快捷地点：一组预置地点按钮。
- 最近搜索：本地持久化的历史关键词按钮和清空历史按钮。
- 地图提示：用于提示当前状态和可用能力。
- 地图展示区：腾讯地图容器和右侧详情卡。
- 详情卡：名称、地址、坐标、当前状态、复制按钮、外链按钮、静态预览、候选列表。

## 4. 运行链路

地图页的运行链路是前端直连腾讯服务，不经过后端业务 API。

1. 用户访问 /place。
2. [app/backend/modules/pages/routes.py](app/backend/modules/pages/routes.py#L6) 通过 send_from_directory 返回 [app/web/place.html](app/web/place.html#L890)。
3. 页面加载 [app/web/theme.css](app/web/theme.css#L841) 中的共享样式。
4. 页面脚本检测 MAP_KEY 是否可用，若可用则加载腾讯地图 JS API 并初始化地图容器。
5. 用户输入关键词或点击快捷地点后，页面先尝试腾讯 WebService place search。
6. 若腾讯返回空结果或失败，再调用本地校园 POI 库兜底。
7. 若腾讯返回候选且本地也有匹配，则合并结果，优先展示本地校园点位。
8. 渲染结果、更新地图中心、更新标记、更新静态预览、更新外链和复制按钮状态。

## 5. place.html 详细拆解

### 5.1 关键常量和数据

- [app/web/place.html](app/web/place.html#L972) 定义了最近搜索的 localStorage key，当前为 bit_place_search_history_v1。
- [app/web/place.html](app/web/place.html#L981) 定义了 QUICK_PLACES 和 LOCAL_POIS 的数据区，其中 LOCAL_POIS 是整个校园地点兜底表。
- [app/web/place.html](app/web/place.html#L1545) 定义了 MAP_KEY。当前地图页会把这个 key 同时用于腾讯地图 JS API、WebService place search 和静态图接口。

LOCAL_POIS 现在已经覆盖了较多校园常用点位，包括：

- 中关村校区的图书馆、校医院、教学楼、食堂、快递服务中心、校门、宿舍与地铁站。
- 良乡校区的徐特立图书馆、门诊部、文化体育中心、教学区、食堂、快递点、宿舍、校门与地铁站。
- 这些条目通常同时配置 title、aliases、campus、tags、location、ad_info 和 source，便于后续匹配、排序和展示。

### 5.2 本地检索逻辑

本地检索相关函数位于以下位置：

- 关键词标准化：[app/web/place.html](app/web/place.html#L1580)
- 关键词变体展开：[app/web/place.html](app/web/place.html#L1586)
- 校区推断：[app/web/place.html](app/web/place.html#L1613)
- 标签推断：[app/web/place.html](app/web/place.html#L1626)

这部分逻辑的作用是把用户输入的关键词转成可比较的标准形式，然后用标题、别名、地址、校区和标签进行综合打分。它的设计目标是服务校园语境，而不是做完全通用的地理搜索。

### 5.3 地图初始化和兜底

地图初始化和无 key 兜底相关逻辑在这里：

- 地图空壳页：[app/web/place.html](app/web/place.html#L1785)
- 腾讯地图脚本加载：[app/web/place.html](app/web/place.html#L1814)
- 地图初始化：[app/web/place.html](app/web/place.html#L1821)

这里有几个重要设计点：

- 如果 MAP_KEY 无效，页面不会假装地图可用，而是直接进入说明模式。
- 如果腾讯静态图被浏览器拦截，页面会回退到本地 SVG 预览，避免页面空白。
- 地图容器只负责显示和定位，真正的候选计算不依赖地图对象本身。

### 5.4 结果列表和结果合并

结果渲染和合并逻辑位于以下位置：

- 候选列表渲染：[app/web/place.html](app/web/place.html#L1874)
- 去重键生成：[app/web/place.html](app/web/place.html#L1895)
- 在线与本地结果合并：[app/web/place.html](app/web/place.html#L1905)
- 单个候选应用：[app/web/place.html](app/web/place.html#L1934)
- 候选结果统一回填：[app/web/place.html](app/web/place.html#L1957)

当前实现已经不是“在线有结果就直接接受”的单一路径，而是：

- 先看腾讯是否有候选。
- 如果本地校园库也有匹配，就把本地结果放在前面，再保留腾讯结果。
- 合并时按标题、地址、区县和坐标做去重，避免同一地点重复出现。
- 这样做的目的，是优先保证校园语境下的准确性。

### 5.5 空状态和提示管理

空状态和提示文案的控制点位于：

- 状态重置：[app/web/place.html](app/web/place.html#L1721)
- 空结果状态：[app/web/place.html](app/web/place.html#L1736)

这里的重点是两个地方：

- 清空结果时，页面会恢复到等待查询状态，不保留旧候选、旧坐标、旧按钮状态。
- 无结果时，地图提示会明确说明当前关键词没有命中在线或本地结果，不会再沿用上一次成功查询的提示。

交互绑定包括：

- 点击查询按钮触发搜索。
- 按回车触发搜索。
- 点击清空结果重置详情和提示。

## 7. 已验证行为

以下是当前已经在浏览器里验证过的行为结论：

- 搜索“静园A”时，页面会返回 2 个候选，且第 1 个是本地良乡校区静园 A，第 2 个是腾讯返回的同名外部候选。
- 搜索“北京理工大学良乡校区”时，页面会返回 5 个候选，并优先展示本地校园点位。
- 搜索乱码关键词“QWERTYZZZ999”时，页面会进入空结果态，候选列表清空，提示文案会重置为当前状态。
- 点击复制坐标后，状态会显示“坐标已复制”。
- 点击复制地址后，状态会显示“地址已复制”。
- 点击结果列表中的第 2 个候选后，页面会切换到对应结果，并更新外链和详情。
- 点击清空结果后，名称、坐标、结果列表和按钮状态会全部恢复初始态，地图提示也会回到“输入关键词后即可查看候选地点”。
- 点击清空历史后，最近搜索列表会被清空。

## 8. 当前已知边界

这部分是地图功能的真实边界，后续排障时可以直接参考。

- 腾讯 place search 并不保证每个校园关键词都能返回校内点位。有时它会返回 0 条结果，有时会返回同名但不在校园内的地点。
- 当前页面采用的是“腾讯在线结果加本地校园库”的混合策略，而不是纯在线策略。
- 当前在线搜索请求使用的是 boundary=region(中国,0) 的查询方式。此前尝试过 nearby 和更细的 region 边界，但测试中要么返回 348 参数错误，要么仍然是空数组。
- 腾讯静态图在部分浏览器自动化场景中可能被拦截，所以页面保留了本地预览回退。
- 地图页不依赖 common.js，也没有独立的后端地图查询 API。如果后续要把查询逻辑迁到后端，需要重新设计接口和返回协议。

## 9. 排障建议

如果后续地图功能出问题，建议按下面顺序排查：

1. 先看 [app/web/place.html](app/web/place.html#L1545) 的 MAP_KEY 是否仍然有效。
2. 再看 [app/web/place.html](app/web/place.html#L1977) 的 searchPlace 是否还能正常发起请求。
3. 如果在线结果为空，先确认本地 POI 是否已经覆盖该关键词，再确认 mergeSearchResults 是否把本地结果放到了前面。
4. 如果页面提示串台，优先检查 [app/web/place.html](app/web/place.html#L1721) 和 [app/web/place.html](app/web/place.html#L1736) 的状态重置逻辑。
5. 如果静态图不显示，检查 updatePreview 和 setPreviewFallback 是否仍然在生效。

## 10. 后续扩展建议

如果你后面还想继续增强地图功能，优先级建议如下：

- 继续补全 LOCAL_POIS，尤其是宿舍楼、教学楼、食堂、快递点和门区。
- 继续优化 searchLocalPois 的评分规则，让同类地点的排序更稳定。
- 如果腾讯开放更合适的校园 POI 查询方式，再评估是否把 boundary 策略改成更精细的区域约束。
- 如果希望把地点查询复用到别的页面，可以把本地检索和结果合并逻辑抽成独立模块，再由 place.html 调用。

## 11. 一句话总结

当前地图页的定位是：腾讯在线检索负责尽可能返回候选，本地校园 POI 负责把结果拉回校园语境，页面再负责把候选、提示、地图、复制、历史和清空这些交互全部串起来。
